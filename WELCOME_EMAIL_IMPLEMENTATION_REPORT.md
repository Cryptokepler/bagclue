**Imports agregados:**


**Service role client:**
```typescript

  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
)
```
- Necesario para query `customer_profiles` desde servidor (bypass RLS)

**Función helper agregada:**
```typescript
async function checkAndSendWelcomeEmail(): Promise<void>
```

**Lógica:**
1. Obtener usuario actual con `supabaseCustomer.auth.getUser()`
2. Query `customer_profiles` con service role: `SELECT created_at, name WHERE user_id = X`
3. Calcular `minutesSinceCreation = (now - created_at) / 60`
4. Si `minutesSinceCreation < 5`:
   - Log: "New user detected, sending welcome email"
   - Call `sendWelcomeEmail({ to, customerName })`
   - `.catch()` para non-blocking error handling
5. Si `minutesSinceCreation >= 5`:
   - Log: "Existing user, skipping"
   - Return sin enviar email
6. Si profile no existe:
   - Log: "Profile not found, skipping"
   - Return sin error

**Integración en flujos:**

**OAuth Flow (Google):**
```typescript
// Después de exchangeCodeForSession SUCCESS
checkAndSendWelcomeEmail().catch(err => {
  console.error('[Welcome Email] Failed in OAuth flow:', err.message)
})
// Continue con redirect
```

**Magic Link Flow:**
```typescript
// Después de verifyOtp SUCCESS
checkAndSendWelcomeEmail().catch(err => {
  console.error('[Welcome Email] Failed in magic link flow:', err.message)
})
// Continue con redirect
```

**Características de seguridad:**
- Fire-and-forget: `.catch()` previene que email failure bloquee redirect
- Non-blocking: usuario continúa a /account incluso si email falla
- Safe logging: no imprime secrets, tokens, passwords
- Error handling: todos los catches loguean pero no lanzan

## LIMITACIÓN CONOCIDA — ANTI-DUPLICADOS

### Problema
El approach de `created_at <5 min` NO previene duplicados si el callback se ejecuta dos veces dentro de 5 minutos.

**Escenario edge case:**
1. Usuario hace click en magic link
2. Callback se ejecuta → envía welcome email
3. Usuario hace click OTRA VEZ en el mismo magic link dentro de 5 min
4. Callback se ejecuta OTRA VEZ → envía welcome email OTRA VEZ

### Probabilidad
**Muy baja** — Requiere:
- Usuario hace click dos veces en magic link rápidamente
- O callback falla y se reintenta automáticamente
- Y ambas ejecuciones ocurren <5 min desde signup

### Soluciones Posibles

#### OPCIÓN A: Aceptar riesgo (recomendado para MVP)
- Probabilidad muy baja
- Impacto bajo (usuario recibe 2 emails)
- No requiere cambios adicionales
- Monitorear logs para detectar si ocurre

#### OPCIÓN B: Agregar flag en DB (más robusto)
**Migración:**
```sql
ALTER TABLE customer_profiles 
ADD COLUMN welcome_email_sent BOOLEAN DEFAULT FALSE;

CREATE INDEX idx_customer_profiles_welcome_sent 
ON customer_profiles(welcome_email_sent);
```

**Lógica actualizada:**
```typescript
if (minutesSinceCreation < 5 && !profile.welcome_email_sent) {
  await sendWelcomeEmail(...)
  
  // Mark as sent
  await supabaseAdmin
    .from('customer_profiles')
    .update({ welcome_email_sent: true })
    .eq('user_id', user.id)
}
```

**Ventajas:**
- ✅ Previene duplicados de forma confiable
- ✅ Idempotente (múltiples ejecuciones safe)
- ✅ Auditable (sabemos quién recibió email)

**Desventajas:**
- ❌ Requiere modificar DB schema (restricción actual)
- ❌ Requiere nueva migración
- ❌ Deploy más complejo

#### OPCIÓN C: Cache/Redis con TTL
**Lógica:**
```typescript
const cacheKey = `welcome_email_sent:${user.id}`
if (await redis.exists(cacheKey)) {
  console.log('Welcome email already sent (cache hit)')
  return
}

await sendWelcomeEmail(...)
await redis.setex(cacheKey, 300, '1') // TTL 5 min
```

**Ventajas:**
- ✅ No modifica DB schema
- ✅ TTL automático
- ✅ Performance óptimo

**Desventajas:**
- ❌ Requiere Redis (no disponible actualmente)
- ❌ Más complejo de configurar

### Recomendación
**Para MVP/Launch:** OPCIÓN A (aceptar riesgo)  
**Para producción madura:** OPCIÓN B (flag en DB)
